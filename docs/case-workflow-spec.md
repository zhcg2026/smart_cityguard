# 案件流程规格（定稿）

> 依据 2026-05-20 业务确认整理；**P0 已于 2026-05-20 落地**。与代码冲突时先改本文再改实现。  
> 进度摘要见 `docs/PROJECT_PROGRESS.md` §2026-05-20。  
> 关联：`CaseStatusConstant`、`case_flow_record`、`verify_task`（库表注释为核实）、`check_task`（库表注释为核查）。

## 1. 术语（业务口径）

| 业务词 | 含义 | 触发条件 | 执行人 | 反馈对象 |
|--------|------|----------|--------|----------|
| **核查** | 立案前补全现场信息 | 信息不完整、无照片等 | 距案件位置最近的采集员 | 受理员 |
| **核实** | 结案前确认现场是否已处置到位 | 信息不完善、照片看不清等 | 距案件位置最近的采集员 | 受理员 |
| **批转** | 推进六步大流程 | 本环节判断通过 | 当前岗位 | 下一环节责任人 |
| **回退** | 修正，不推进大流程 | 非本环节职责 / 处置不达标等 | 当前岗位 | 上一环节或指定环节 |

**作废**：不符合立案条件 → 案件结束（`not_accepted` / 作废），不再流转。

**命名对齐（技术债）**：库表 `verify_task` 注释为「核实」、`check_task` 为「核查」，与部分 Java 方法名（`verifyCase`/`checkCase`）曾混用。实现时**以本节业务词为准**，P1 可做 API/常量别名整理。

## 2. 六步闭环（对外）与细状态（对内）

对外固定六步：**上报 → 立案 → 派遣 → 处置 → 反馈 → 结案**。  
对内保留细状态，并增加 **`stage`**（或等价展示字段）映射六步，供列表/详情展示。

| 对外阶段 | 典型细状态（示例） |
|----------|-------------------|
| 上报 | `reported`, `pending_register`, `pending_verify`（核查中） |
| 立案 | 受理员操作中；作废 `not_accepted` |
| 派遣 | `pending_dispatch`, `returned`（回退修正中） |
| 处置 | `pending_handle`, `handling`, `handle_finish` |
| 反馈 | `pending_check`（部门已确认 → 派遣员把关 → 可批转受理员） |
| 结案 | `closed`；核查/核实任务进行中可视为反馈子状态 |

## 3. 主干批转链（默认路径）

```
采集员上报 → pending_register（公共池）
  ├─ [可选] 受理员下发核查 → 采集员核查 → 回受理员 → 再立案/作废
  ├─ 作废 → not_accepted（结束）
  └─ 立案并批转派遣员 → pending_dispatch（current_handler=派遣员）
        ├─ [回退] 非我局 → returned → 受理员（优先立案人）
        └─ 派遣至处置部门 → pending_handle（current_handler 空，部门待指派）
              ├─ [回退] 非本单位 → 先撤销指派 → returned → 派遣员
              ├─ 指派 HANDLER → handling
              │     ├─ [回退] 处置前 → 部门（移动端，理由必填）
              │     └─ 提交处置 → handle_finish
              ├─ [回退] 部门认为未到位 → handling（原 HANDLER 再处置）
              └─ 部门确认批转 → pending_check（current_handler=派遣员）
                    ├─ [回退] 派遣员认为未到位 → pending_handle（原部门返工，保留全过程数据）
                    ├─ 批转受理员 → pending_check（current_handler=指定受理员）
                    │     ├─ [可选] 下发核实 → 采集员核实 → 回受理员
                    │     ├─ [回退] 不达标 → pending_dispatch（current_handler=当班派遣员，原部门再处置）
                    │     └─ 结案（须满足立结案标准必选）→ closed
                    └─ [约束] 部门刚回退至派遣员时：派遣员仅可回退受理员或再派部门，不可批转受理员
```

## 4. 回退规则明细

### 4.1 派遣员 → 受理员（非我局）

| 项 | 规则 |
|----|------|
| 状态 | `returned`（回退），**不清空**已填立案/派遣/部门等字段 |
| 接收人 | **优先** `register_operator_id`（立案受理员）；不在班可指定其他受理员 |
| 理由 | 必填 |
| 权限 | 回退后派遣员在该案上仅保留查看/再派等规则见 §5 |

### 4.2 派遣员 → 处置部门（处置不达标）

| 项 | 规则 |
|----|------|
| 时机 | 处置部门已反馈（`deptConfirm` 之后）、`pending_check` 且 `current_handler`=当班派遣员 |
| 状态 | `pending_handle`（返工），**原 `handle_dept_id` 不变** |
| 数据 | **保留**全部过程数据（流转、附件、处置记录） |
| 理由 | 必填（描述需完善点） |
| 部门 | 根据派遣员描述继续完善处置 |

### 4.3 处置部门 → 派遣员（非本单位）

| 项 | 规则 |
|----|------|
| 时机 | `pending_handle` |
| 前置 | 若已指派 HANDLER，须先 **撤销指派** 再回退 |
| 状态 | `returned`，`current_handler`=原派遣员（当班派遣员逻辑见 §6） |
| 理由 | 必填 |

### 4.4 处置部门 → 处置人员（处置不达标）

| 项 | 规则 |
|----|------|
| 时机 | 已收到 HANDLER 提交（`handle_finish`），部门初审不通过 |
| 状态 | `handling`，`current_handler`=原 HANDLER |
| 理由 | 必填 |
| 数据 | 保留上一轮处置材料 |

### 4.5 处置人员 → 处置部门

| 项 | 规则 |
|----|------|
| 时机 | **处置提交前**（`handling`），提交后不可撤回 |
| 端 | 移动端（`collector-app` 处置端） |
| 理由 | 必填 |

### 4.6 受理员 → 派遣员（结案前处置不达标）

| 项 | 规则 |
|----|------|
| 时机 | 反馈阶段，受理员收案后（`pending_check` + `current_handler`=该受理员） |
| 接收派遣员 | **不绑定**立案时派遣员，**当班** `current_handler` 派遣员 |
| 状态 | `pending_dispatch` 或返工语义等价为 **原部门再处置**：实现落为 `pending_handle` + 保持 `handle_dept_id`，`current_handler`=当班派遣员（与 §4.2 派遣员返工部门一致，见实现备注） |
| 理由 | 必填，不需勾选标准不达标项 |
| 与核查区别 | 此处为**返工**，不是下发采集员「核实」任务 |

**实现备注**：受理员「不达标回派遣员→原部门再处置」与 §4.2 汇合：派遣员将案件派回同一 `handle_dept_id`，部门再指派/原人再处置。

## 5. 回退接收方权限（状态机约束）

> 回退接收方**不能执行下一阶段的「主推进」操作**，只能做本环节允许的修正或再批转。

| 场景 | 接收方 | 禁止 | 允许 |
|------|--------|------|------|
| 部门回退至派遣员后 | 派遣员 | 直接批转受理员结案 | 回退受理员、再派遣至部门（含原部门） |
| 派遣员回退至受理员后 | 受理员 | 派遣、结案 | 立案批转、作废、下发核查 |
| 受理员回退至派遣员后 | 派遣员 | 批转受理员 | 再派部门、回退受理员 |
| 部门打回 HANDLER 后 | HANDLER | 批转、结案 | 再处置提交 |
| 公共池案件 | 全体受理员 | — | 先立案/作废者认领 |

**结案**：仅 `pending_check` 且 `current_handler`=被派遣员指定的受理员；须通过立结案标准必选校验。

**派遣员**：不绑定立案时人选；以 **`current_handler`** 为准，支持当班转办。

## 6. 多账号与归属

| 规则 | 说明 |
|------|------|
| 受理员公共池 | `register_operator_id` 为空 + `pending_register` / `reported` / 核查中 |
| 认领 | **维持现状**：首次立案或作废成功写入 `register_operator_id` |
| 谁立案谁负责 | 立案人全程可查（「我立案的案件」）；**收尾受理员**可由派遣员指定任意受理员 |
| 派遣员 | 不全程绑定；按当班 `current_handler` |
| 采集员任务 | 核查/核实均派**距案发位置最近**的采集员（P1 实现选址算法，可先责任网格+距离） |

## 7. 核查与核实（任务，非每案必走）

### 7.1 核查（立案前）

- 受理员在待立案案件上操作「发送核查」。
- 系统创建 **`check_task`**（核查任务表），指派最近采集员。
- 案件进入核查中状态（建议 `pending_verify` 或子状态 + `check_task_id`）。
- 采集员移动端提交核查结果与照片 → 受理员待办可见 → 再决定立案/作废。

### 7.2 核实（结案前）

- 受理员在反馈/待结案阶段操作「发送核实」（仅信息不完善、照片不清等）。
- 系统创建 **`verify_task`**（核实任务表），指派最近采集员。
- 采集员现场核实是否已处置 → 反馈受理员 → 受理员再结案或回退派遣员返工。
- **不是**受理员菜单里的「核实通过/不通过」旧逻辑；旧 `checkCase` 达标返工应改为 **回退派遣员**。

### 7.3 与结案关系

- **结案**：独立操作，立结案标准**必选**。
- **核实任务**：可选支线，完成后仍须受理员主动结案。
- 不达标返工：走 §4.6，**不是**核实任务。

## 8. 立结案标准

| 环节 | 规则 |
|------|------|
| 立案 | 未选标准 / 必选条件未满足 → **不允许立案** |
| 结案 | 同上 → **不允许结案** |
| 回退 | 仅填文字原因，不要求勾选不达标项 |

## 9. 流转留痕（`case_flow_record`）

P0 起新记录建议统一：

| 字段 | 说明 |
|------|------|
| `operate_type` | `forward` / `return` / `cancel` / `close` / `assign` / `revoke_assign` |
| `operate_opinion` | 回退**必填**；批转建议填 |
| `receiver_id` / `receiver_name` | 批转/回退目标人（及部门） |
| `node_code` / `node_name` | 对应当前细状态 + 业务动作名 |

保留历史记录只增不改。状态迁移与 `operate_type` 在服务端集中校验（状态机表或 `CaseFlowGuard`）。

## 10. 端侧能力矩阵

| 能力 | 管理端 admin-web | 移动端 collector-app |
|------|------------------|----------------------|
| 受理员立案/作废/核查/核实/结案/回退 | ✓ | 视产品（建议受理员以 PC 为主，可二期） |
| 派遣员派遣/回退/批转受理员 | ✓ | 可选二期 |
| 部门指派/确认/回退 | ✓ | 可选 |
| 处置人员处置/回退部门 | ✓（若需要） | ✓（必须） |
| 采集员核查/核实任务 | — | ✓ |

**原则**：系统应 **PC + 移动** 均可支撑；P0 可先保证 **处置回退 + 主干批转** 双端，其余岗位移动能力按 P1 排期。
