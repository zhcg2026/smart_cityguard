# 申诉逻辑重构计划

## 现状分析

后端 `TimeoutAppealServiceImpl.submit()` 当前仅允许 `ROLE_DEPT` 提交，状态直接为 `pending_dispatcher`。

用户要求的完整流程：
- **处置人员(HANDLER)** 提交 → **处置部门(DEPT)** 审核 → **派遣员(DISPATCHER)** 审核 → **受理员(ACCEPTOR)** 审核
- **处置部门(DEPT)** 直接提交 → **派遣员** 审核 → **受理员** 审核

## 后端改动

### 1. `TimeoutAppealConstant.java` — 新增状态

```java
public static final String STATUS_PENDING_DEPT = "pending_dept";  // 新增
```

### 2. `TimeoutAppealServiceImpl.java` — submit 方法改造

- `assertRole` 改为允许 `ROLE_DEPT` 和 `ROLE_HANDLER`
- HANDLER 提交时：
  - 状态设为 `pending_dept`
  - `applyUserId` = 当前用户 ID
- DEPT 提交时：
  - 状态设为 `pending_dispatcher`（保持不变）
- 两个角色都需要验证：案件属于本部门、已结案、处置超时、未重复申诉

### 3. `TimeoutAppealServiceImpl.java` — 新增 deptReview 方法

```java
public AppealApply deptReview(TimeoutAppealReviewRequest request, LoginUser user) {
    // 仅 ROLE_DEPT 可操作
    // 状态必须为 pending_dept
    // 通过 → 状态变为 pending_dispatcher
    // 驳回 → 状态变为 rejected
}
```

### 4. `TimeoutAppealController.java` — 新增接口

```java
@PostMapping("/dept-review")
public Result<AppealApply> deptReview(@RequestBody TimeoutAppealReviewRequest request)
```

### 5. `TimeoutAppealServiceImpl.java` — list 方法调整

移动端列表按角色过滤：
- HANDLER：只看自己提交的（`applyUserId = currentUserId`）
- DEPT：本部门所有申诉（现有逻辑）
- DISPATCHER/ACCEPTOR：现有逻辑

### 6. `TimeoutAppealServiceImpl.java` — getDetail 调整

- 新增 `canDeptReview` 字段（ HANDLER 提交的 pending_dept 状态，且用户是 DEPT 角色）

## 前端改动

### 7. `src/api/appeal.js` — 新增 deptReview API

```js
export function deptReviewTimeoutAppeal(data) {
  return request({ url: '/appeal/timeout/dept-review', method: 'post', data })
}
```

### 8. `src/views/handle/HandleDetail.vue` — 申诉入口调整

当前：仅 `handleTimer.overdue && caseStatus === 'handling'` 显示按钮
改为：案件已结案（`closed`/`forced_close`）+ 处置阶段超时 + 本部门案件 + 未申诉过
- 处置人员和处置部门角色都可见

### 9. `src/views/appeal/list.vue` — 申诉列表页重构

底部 tab "申诉" 打开后的列表页：
- 处置人员：显示自己提交的申诉
- 处置部门：显示本部门所有申诉（含待审核的）
- 列表项显示：案件编号 + 申诉状态 + 提交时间
- 点击进入申诉详情

### 10. `src/views/appeal/AppealDetail.vue` — 申诉详情页增加审核操作

- 处置部门角色 + 状态 `pending_dept` 时：显示"通过"/"驳回"按钮
- 驳回时弹出输入框填写意见
- 调用 `deptReviewTimeoutAppeal` API

## 执行顺序

1. 后端：Constant 加状态 + submit 改造 + deptReview 方法 + Controller 接口
2. 前端：API + 列表页 + 详情页审核操作 + HandleDetail 入口条件调整

## 验证方式

1. 编译后端：`mvn compile -pl smart-cityguard-appeal -am -q`
2. 编译前端：`npx vite build`
3. 测试流程：
   - 处置人员登录 → 打开已超时结案的案件 → 点击"申请超时申诉" → 填写描述提交 → 状态为"待部门审核"
   - 处置部门登录 → 底部"申诉"tab → 看到待审核的申诉 → 点击详情 → 审核通过 → 状态变为"待派遣员审核"
   - 派遣员/受理员在管理端审核
